Tracking Changes
Questions
How do I record changes in Git?

Objectives
Go through the modify-add-commit cycle for one or more files.
Explain where information is stored at each stage of that cycle.
Distinguish between descriptive and non-descriptive commit messages.
Steps
Create a file named thesis.txt
Open the newly created file and type This is my thesis version 1.0
Type git status , you should see now the new change as untracked change. What does that mean?
We need to keep tracking this file from now on, let’s add it to the git tracker
git add thesis.txt
Bash
Now re-type git status, noticed the difference? you will see all your tracked files here.
This should be our first commit with that file. Let’s commit our tracked change.
git commit -m "Initial commit"
Bash
Try now to modify the file content and change the version to 2.0, and type git status again. Run git status and notice the output.
What is the history of your edits? git log
Where are my changes? git diff
Commit the new edit and name your commit "My thesis version 2.0"
Staging Area
When you type git add, git takes a snapshot of the tracked changed and put these changes in the staging area. That means, you can git add many files and commit them once. Please note that when you type git add on a file, this file will be kept always in git tracking system and appears in the git status. However, it will not be committed to git unless you stage it first using git add.
You can limit git diff to view the staged files only by adding --staged flag.
Exploring History
View the past history of the thesis file using git log -p thesis.txt now you can see the previous commits alongside their hashes.
Recovering the old version of the thesis
git checkout <commit_hash> <file_name> # git checkout 9080sd80a9s90asd870udaj thesis.txt
Bash
Now view the file using cat thesis.txt you will see the old file.
You can restore this file to this version using git restore --staged thesis.txt then commit as a new change.
Branching from a previous commit
Create a branch from commit hash
git branch branchname <sha1-of-commit>
Bash
For now, let’s use reference commit (move 2 commits back and create the branch)
git branch patch1 HEAD~2 # Then checkout git checkout patch1 # modify the thesis file and add a new line echo "this is a new line" >> thesis.txt # Stage the file git add thesis.txt # Commit the changes git commit -m "change inside patch"
Bash
Now we are ready to merge the changes in the patch1 branch to our main branch.
# Checkout the main branch git checkout main # Merge git merge patch1

Put it all together
GitHub
Questions
How do I share my changes with others on the web?
Objectives
Explain what remote repositories are and why they are useful.
Push to or pull from a remote repository.
Version control really comes into its own when we begin to collaborate with other people. We already have most of the machinery we need to do this; the only thing missing is to copy changes from one repository to another.
Systems like Git allow us to move work between any two repositories. In practice, though, it’s easiest to use one copy as a central hub, and to keep it on the web rather than on someone’s laptop. Most programmers use hosting services like GitHub, Bitbucket or GitLab to hold those main copies; we’ll explore the pros and cons of this in a later episode.
Let’s start by sharing the changes we’ve made to our current project with the world. To this end we are going to create a remote repository that will be linked to our local repository.
Connect using SSH
Create an SSH key pair
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"
Bash
Press enter until the end. No need for passphrase.
Verify the generation by typing ls -al ~/.ssh
Copy the public key to your GitHub account. Settings > SSH and GPG keys > SSH Keys > Add key
Verify the connection by typing ssh -T git@github.com
1. Create a remote repository
Name the repo: my_thesis
Connect the local repo the remote one
git remote add origin git@github.com:<github_username>/<repo_name>.git
verify git remote -v
Now you can use the command git push after committing your changes to reflect the changes on the remote GitHub repo.
